# PPTX completion + EOS overhaul + Network diagram — Design

> Status: Approved (brainstorming) — 2026-06-24. Combined spec; decomposes into three independent implementation plans (A/B/C).

## Goal

Finish three gaps left after the patlas Proxmox fork:

1. **PPTX** — the generated deck still says "vatlas" and is missing the three new Proxmox-native metrics.
2. **EOS** — OS End-of-Support is Windows-centric, has no Proxmox VE (node) coverage, and mis-handles Proxmox host versions.
3. **Network diagram** — the Proxmox `.zip` bundle ships a `network-diagram.svg` that patlas extracts and then discards, while the inherited Network view is empty for Proxmox.

## Non-goals

- HTML report changes for the PPTX metrics (deck-only for Snapshot Sprawl / Storage Content / Cluster Health).
- EOS bucket/forecast redesign — keep the existing 3/6/9/12-month windows and the EOS view layout.
- Renaming internal code comments / type-doc "vatlas" mentions (not user-facing — YAGNI).

## Global constraints (inherited, binding)

- Engines pure (no React/DOM/Zustand; Zod only at the parser boundary). Single `useMemo` in `useEstateView`; slices compose in the one `buildEstateView` pass.
- Branded units; `null` = not-derivable (never 0).
- i18n keys in all 4 locales (`keyParity`); no VMware tokens in values (`terminology`); no editorial verbs; no pre-formatted numbers.
- Privacy (PAR-05): no egress of dataset bytes; no `localStorage` of rows; `xlsx`/`fflate` worker-only. The network SVG is dataset-derived → in-memory only, never egressed.
- **EOL/vendor dates are technical facts: validate against an authoritative source (endoflife.date) at implementation time — never write them from memory.**
- PPTX visuals use **native pptxgenjs shapes/text, not rasterized SVG** (resvg-no-font: rasterized SVG text vanishes).
- No NUL-byte corruption in `.ts` (prefer targeted edits; verify `tr -dc '\000'` is 0).
- Commit prefix per plan; signed commits.

## Plan A — PPTX: branding + three new slides

### A1. Branding fix (user-visible "vatlas")

- `engines/export/pptx/slides/titleSlide.ts` — the footer `t('generated', 'Generated by vatlas …')` default → "patlas"; update the `pptx`-namespace `generated` value in all 4 locale files if present.
- `hooks/useExport.ts` — export filename `vatlas_<vc>_<iso>.<ext>` → `patlas_…`.
- `components/inventory/DataTable.tsx` — CSV filename `vatlas-<kind>-<date>.csv` → `patlas-…`.
- `pwa/registerSW.ts` — fix the stale `/vatlas/` comment; confirm the SW scope resolves from `import.meta.env.BASE_URL` (`/patlas/`), not a hardcode.
- Test: a unit test on the export-filename helper asserting the `patlas_` prefix; grep gate that no user-facing export string contains "vatlas".

### A2. Three new deck slides

Each is a pure builder in `engines/export/pptx/slides/`, reading its existing `EstateView` slice (no new engine work), modeled on `rightSizingSlide`/`monsterSlide` (KPI band + native table; native shapes/text only). Registered in the deck builder after `monsterSlide`, before `contentionAnnex`. New `pptx`-namespace i18n keys in all 4 locales.

| Slide | Source slice | KPIs | Table(s) | Emits when |
|---|---|---|---|---|
| `snapshotSprawlSlide` | `view.snapshotSprawl` | snapshots, guests w/ snapshots, total GiB, oldest days | checkpoints, oldest-first | `snapshotSprawl.count > 0` |
| `storageContentSlide` | `view.storageContent` | total GiB, files, content types, backups | by-content-type; by-storage | `storageContent.fileCount > 0` |
| `clusterHealthSlide` | `view.clusterHealth` | quorum, fencing, HA resources, backup jobs | HA services; backup jobs | `ha.services.length > 0 || ha.managedCount > 0 || backups.jobCount > 0` |

- Tests: per-slide unit test asserting emit/omit on the boundary condition; the builder includes them in order. **Visual verification**: render the deck → PDF → read, confirming the three slides render with legible native text (per the "verify deck visually" rule).

## Plan B — EOS overhaul

### B1. Catalogue refresh (`engines/eos/catalogue.json`)

Expand `products` with Proxmox-common guest OSes, EOL dates **validated against endoflife.date** at build time; bump `lastVerified`:

- Debian 10, 11, 12, 13; Ubuntu 16.04–24.04 LTS; RHEL 7–9 + Rocky/Alma 8–9; SUSE SLES 12–15 (+ openSUSE Leap); keep Windows Server entries.
- Schema (`catalogueSchema.ts`) unchanged. Test: schema validation + a `lastVerified` freshness assertion + spot-checks of a few known EOL dates.

### B2. Proxmox VE (node) EOL — replace the ESXi path

The Proxmox adapter stores the PVE version in `VHostRow.esxVersion` (from the Nodes "PVE Version" column).

- Add **Proxmox VE** entries (PVE 6, 7, 8 — validated EOL dates) to the catalogue.
- New `engines/eos/classifyPve.ts` — parse `pve-manager/8.2.2/…` or `8.2.2` → `major.minor` → PVE release → EOL bucket; replaces `classifyEsxi.ts` (removed — patlas has no ESXi).
- Rename the host bucket end-to-end: `EosProjection.esxi → .nodes` (type in `types/estate.ts`), `bucketEos.ts` (`esxiCounts → nodeCounts`, `classifyEsxi → classifyPve`), `EMPTY_EOS`, `EosView`, `eosSlide` labels ("ESXi" → "Node"/PVE). This rename is the coupling point — touches the typed slice + all its consumers + tests.
- Tests: `classifyPve` table test (real PVE version strings → buckets); `bucketEos` node-bucket test; updated `EosView`/`eosSlide`/`EMPTY_EOS` tests.

### B3. Normalization (`engines/eos/normalizeOs.ts`)

Tune to match Proxmox `OS name`/`OS version` strings and LXC template names (e.g. `debian-12-standard`, `ubuntu-22.04-standard`) so more guests classify instead of "unknown". Driven by a table test seeded from real fixture OS strings.

## Plan C — Network diagram (the bundled SVG)

### C1. Carry the SVG

- `engines/parser/extractZip.ts` already returns `networkSvg`. Stop discarding it in `parser.worker.ts`: add `networkSvg: string | null` to the assembled `Snapshot` (and the worker `ParseRequest`/response types). Bare `.xlsx` (no bundle) ⇒ `null`.
- Add `networkSvg?: string | null` to the `Snapshot` type as an **optional** field (like `rawReleased?` — avoids updating every existing `Snapshot` test builder; the worker always sets it, `null` when no bundle). It does NOT go through `mergeSnapshotsToEstate`/`EstateView` (it's a per-active-snapshot asset); the view reads it from the active snapshot directly via `selectActiveSnapshot`.

### C2. Render safely as the Network view

- `components/network/NetworkView.tsx` — when the active snapshot has `networkSvg`, render it as the Proxmox topology (the existing vSwitch/dvSwitch rollup is empty for Proxmox; show the diagram instead, falling back to the current empty-state when `networkSvg` is null).
- **Security:** render via a **sandboxed `<img>`** whose `src` is a `data:image/svg+xml;…` (or blob) URL built from the SVG string — NOT `dangerouslySetInnerHTML`. An SVG in an `<img>` cannot execute scripts (no XSS from a semi-trusted report file). Blob/data URLs are same-origin (privacy guard intact; no egress). The diagram has no external `<image>`/script, but the `<img>` boundary makes that safe regardless.
- Test: NetworkView renders an `<img>` with an svg data-URI when `networkSvg` is present; falls back to the empty state when null. (jsdom asserts the element/attribute, not pixels.)

### C3. HTML report

- `engines/export/html/renderReport.tsx` — embed the diagram as a `data:image/svg+xml` `<img>` in a "Network" section when present. Safe-by-construction (data-URI img in a static document). Omit the section when null. PPTX deliberately excluded (SVG-text rasterization caveat).

## Decomposition & sequencing

Three independent plans, each its own branch/PR:

- **Plan A (PPTX)** — branding + 3 slides. Low risk; slides are parallelizable. No type changes.
- **Plan B (EOS)** — catalogue + `classifyPve` + the `esxi→nodes` slice rename + normalization. Largest blast radius (typed slice rename); build on its own.
- **Plan C (Network diagram)** — carry `networkSvg` + NetworkView render + HTML report. Touches the parser worker + Snapshot type + NetworkView + report.

Build order suggestion: C (small, self-contained) → A → B (or A and C first since B's data-validation is the slowest). No hard dependency between them; A's `eosSlide` label change overlaps B's rename, so if both run, B owns the `eosSlide` ESXi→Node relabel.

## Testing (all plans)

Unit tests per new function/slide; engines stay ≥75% coverage; i18n `keyParity` + `terminology` green; full `npm run typecheck` + `biome check .` + `vitest run`; visual deck verification for Plan A; no NUL bytes.
