# Testing

How vatlas is tested: the framework, how to run the suite, where tests live, the
coverage gate, when real-browser rendering is required, and how tests run in CI.

For local development setup see [DEVELOPMENT.md](./DEVELOPMENT.md). For the
engines/store/UI tier boundaries that shape what each test exercises, see
[ARCHITECTURE.md](./ARCHITECTURE.md).

## Test framework and setup

| Tool | Version | Role |
|------|---------|------|
| Vitest | `^4.1.6` | Test runner (unit, integration, component) |
| `@vitest/coverage-v8` | `^4.1.6` | V8 coverage provider |
| `@testing-library/react` | `^16.3.2` | Component rendering and queries |
| `@testing-library/jest-dom` | `^6.9.1` | DOM matchers (`toBeInTheDocument`, etc.) |
| `@testing-library/user-event` | `^14.6.1` | Realistic user interaction simulation |
| jsdom | `^29.1.1` | Default test environment |

Configuration lives in `vitest.config.ts`. Key settings:

- `globals: true` — `describe`/`it`/`expect`/`vi` are available without imports.
- `environment: 'jsdom'` — the default DOM environment, pinned to
  `http://localhost/` via `environmentOptions.jsdom.url` so origin-bound APIs
  (`window.localStorage`) work instead of being withheld from an opaque origin.
- `setupFiles: ['./src/test/setup.ts']` — runs once per worker.
- `include` — `src/**/*.{test,spec}.{ts,tsx}` and
  `tests/**/*.{test,spec}.{ts,tsx}`.
- Path aliases (`@`, `@engines`, `@components`, `@store`, `@types`, `@utils`,
  `@hooks`) mirror the app config so test imports match production imports.

`src/test/setup.ts` bootstraps the environment before any test renders:

- Registers `@testing-library/jest-dom/vitest` matchers.
- Imports `../i18n` so `useTranslation()` returns real EN/FR strings, not keys.
- Forwards jsdom's `localStorage`/`sessionStorage` onto the test global (Node 26
  - Vitest 4 + jsdom 29 omit them from `populateGlobal`).
- Provides a deterministic non-zero viewport (`ResizeObserver` stub,
  `offsetHeight`/`offsetWidth`/`getBoundingClientRect` overrides) so the
  `@tanstack/react-virtual` window does not collapse to zero rendered rows under
  jsdom.
- Stubs `window.matchMedia` (jsdom does not implement it; `useTheme` calls it on
  mount).

No setup beyond `npm install` is required to run the suite.

## Running tests

```bash
npm run test          # watch mode (vitest)
npm run test:run      # single CI-mode run (vitest run)
npm run test:coverage # single run with V8 coverage
```

Run a single file, a directory, or filter by test name (pass-through args after
`--`):

```bash
# One file
npx vitest run src/engines/units/converters.test.ts

# One directory (all engine aggregation tests)
npx vitest run src/engines/aggregation

# Filter by test name (substring of describe/it)
npx vitest run -t "sorts the VM rows by provisionedMib"

# Watch a single file while iterating
npx vitest src/components/Chart.test.tsx
```

## Writing new tests

- **Naming.** Test files are colocated next to the code they cover and end in
  `.test.ts` or `.test.tsx` (e.g. `converters.test.ts`,
  `DrSimPanel.test.tsx`). `.spec.*` is also matched by config but the codebase
  uses `.test.*`. Cross-cutting smoke/stress suites live under
  `src/__tests__/` (e.g. `e2e-smoke.test.tsx`, `inventory-stress.test.tsx`).
- **Shared bootstrap.** All environment shims live in `src/test/setup.ts` — do
  not re-stub `matchMedia`, `ResizeObserver`, or layout metrics per file; they
  are already provided.
- **i18n in component tests.** Call `await i18n.changeLanguage('en')` (or `'fr'`)
  in `beforeEach` when asserting on rendered text; i18next is initialized by the
  setup file with bundled resources.
- **Engines are pure.** `src/engines/**` and `src/utils/**` take inputs and
  return values with no React/DOM/store dependency — test them as plain
  functions. Components consume `useEstateView`, never engines directly; mirror
  that boundary in tests (drive UI through the hook/merge path, assert engine
  math in isolation). See [ARCHITECTURE.md](./ARCHITECTURE.md).
- **Synthetic stress fixtures.** Real RVTools fixtures top out at ~249 VMs.
  Performance budgets are proven against a deterministic synthetic ~10k-VM
  workbook generated by `scripts/generate-inventory-10k.mjs`
  (`npm run generate-inventory-10k`). The fixture
  (`src/__fixtures__/rvtools-inventory-10k.xlsx`) is gitignored; stress suites
  regenerate it in `beforeAll` if absent so CI stays self-contained. The
  generator output is byte-identical across runs. `src/__tests__/inventory-stress.test.tsx`
  drives the fixture through the same pure pipeline the worker runs
  (`parseXlsx → parseSnapshot → buildEstateView`) and asserts hard budgets:
  VM sort by `provisionedMib` completes in under 200 ms at 10k, the virtualised
  tree window stays bounded (≪ total leaf count), and CSV export preserves raw
  values and embedded newlines (RFC-4180 quoting) under both light and dark
  themes. Do not widen these budgets to make a run pass — surface the breach.
- **MiB canary.** `npm run generate-mib-canary` produces the unit-conversion
  canary fixture used by `src/engines/parser/canary.test.ts` to guard the
  RVTools-MB-is-MiB invariant.

## Coverage requirements

Coverage uses the V8 provider with `text`, `json`, and `html` reporters. The
gate applies to the pure logic layers only — UI components are intentionally
excluded from the threshold.

| Type | Threshold |
|------|-----------|
| Lines | 75% |
| Functions | 75% |
| Branches | 75% |
| Statements | 75% |

Gated (`coverage.include`) paths:

- `src/engines/**/*.ts`
- `src/utils/**/*.ts`
- `src/engines/units/**/*.ts`
- `src/privacy/**/*.ts`

Excluded: `**/*.d.ts`, `**/*.test.ts`, `**/*.spec.ts`. `npm run test:coverage`
fails if any threshold drops below 75% on the gated paths.

## When to use Vitest Browser Mode

The default jsdom environment does **not** lay out the DOM and does **not**
produce real SVG geometry. Charts in vatlas render through Apache ECharts with
the SVG renderer (`src/components/Chart.tsx`). Assertions about *real* chart
geometry — computed SVG path coordinates, element layout, pixel positions —
cannot be made under jsdom because the produced SVG has no real geometry.

Use **Vitest Browser Mode** (Vitest 4, stable) for any test that must assert on
actual rendered SVG geometry of an ECharts chart. For everything else — engine
math, store behavior, hook logic, component behavior that does not depend on
laid-out chart geometry — the default jsdom environment is correct and faster.
The jsdom layout shims in `src/test/setup.ts` cover virtualisation window
bounds (node counts), not chart vector geometry.

## CI integration

Tests run in the **Deploy to GitHub Pages** workflow
(`.github/workflows/static.yml`), triggered on push to `main`, version tags
(`v*`), pull requests targeting `main`, and manual dispatch.

The `build` job runs the gate sequence on Node 24:

1. Supply-chain check (`scripts/check-supply-chain.mjs`)
2. `npm ci`
3. `npm audit --audit-level=low`
4. OSV-Scanner (lockfile, gated on LOW+)
5. **Type check** — `npm run typecheck`
6. **Lint** — `npm run lint`
7. **Test** — `npm run test:run`
8. **Build** — `npm run build`
9. Bundle-size check (`npm run check:bundle-size`)
10. SBOM generation and upload

The shipped test step is `npm run test:run` (`vitest run`); coverage
(`npm run test:coverage`) is run locally to enforce the 75% gate and is not a
separate CI step in the current workflow. Deployment to
`fjacquet.github.io/vatlas/` happens in a dependent `deploy` job only for pushes
to `main`, gated on the full `build` job passing.
