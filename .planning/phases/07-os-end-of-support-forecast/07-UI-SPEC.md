# Phase 7 — UI Design Contract

> Visual and interaction contract for the OS / ESXi End-of-Support forecast view (bucketed lifecycle counts, bucket→entity drill, catalogue freshness, first-class unknown-OS).
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.
> Authoritative design source: `07-CONTEXT.md <decisions>` D-00..D-12 (LOCKED — not re-opened here).
> The project stack + Midnight Executive palette are locked project-wide in `CLAUDE.md`; no stack/component-library decision is made or re-asked in this phase. There is no `07-RESEARCH.md` at UI-spec time by design (researcher confirms catalogue/granularity facts; this contract is presentation-only).

## Spacing Scale

Declared values (multiples of 4; matches the shipped views — `HostsView`/`PlanningView`/`InventoryView` — verbatim; do not introduce new spacing tokens):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | `gap-1` ViewToggle inner segment gap; `p-0.5` toggle frame padding; `gap-1.5` stat label↔value |
| sm | 8px | `gap-2` bucket-tile label↔count; `mb-2`/`mb-3` sub-headings; freshness line↔caption gap |
| md | 16px | `.panel` padding (`p-4`); intra-panel section stack |
| lg | 24px | `gap-6` page section stack (`flex flex-col gap-6`, verbatim with `HostsView`); bucket-strip row gap |
| xl | 32px | `p-8` view scroll-container padding (`<main className="flex-1 overflow-y-auto p-8">`, verbatim with `HostsView`); `gap-x-8` stat-row column gap |
| 2xl | 48px | Reserved — major section break between the bucket-strip/chart block and the (entity drill / unknown-OS diagnostics) block when both are open |
| 3xl | 64px | Reserved — page-level only; not used inside EOS panels |

Exceptions:

- Bucket tiles are click targets: minimum tile height `h-20` (80px) with full-tile `<button>`/clickable hit area ≥ 44px in both axes (touch-target floor; counts + label comfortably exceed it). The ViewToggle segment (if a 5th segment is added) keeps the shipped `h-10` (40px) button height verbatim — do not invent a bespoke 44px exception for the nav segment; preserve the shipped rhythm (precedent: `06-UI-SPEC` accepted `h-9`/`h-10` toggle heights for visual consistency).
- The drill `DataTable` keeps its shipped `ROW_HEIGHT = 36` virtualized row height verbatim (P3 primitive — not restyled here).

## Typography

Matches shipped vatlas type usage exactly (sizes are Tailwind defaults already in use across `HostsView`/`PlanningView`; 2 weights only):

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body / caption (freshness line, "as of {today}", outdated caption, raw-OS-string rows) | 14px (`text-sm`) | 400 (normal) | 1.5 |
| Label / bucket label / stat label | 14px (`text-sm`) | 600 (semibold) for labels, 400 for the qualifier text | 1.5 |
| Bucket count (the figure on each tile) | 24px (`text-2xl`) | 600 (semibold) | 1.2 |
| Panel / section heading | 20px (`text-xl`) | 600 (semibold) | 1.2 |

Rules:

- Exactly 2 weights: `font-normal` (400) and `font-semibold` (600). No other weights.
- Exactly 4 sizes: 14 / 20 / 24px declared (the `text-xs` 12px caption used elsewhere is NOT introduced for this phase — keep freshness/“as of” at `text-sm` for legibility, matching `HostsView` body text).
- Every numeric value (bucket counts, occurrence counts, reconciliation total) renders in `--font-mono` with `tabular-nums` (verbatim shipped idiom from `HostsView` `<tbody className="font-mono tabular-nums">`). Dates are rendered via the locale-aware `src/utils/format.ts` helpers — never pre-formatted in i18n strings, never raw `toLocaleDateString` inline.
- The verbatim original RVTools OS string in the unknown-OS list renders in `font-mono` (it is preserved-verbatim data per D-12, not prose) with the row text wrapped/`break-all`-safe like the shipped `HostsView` `hostName` cell.

## Copywriting Contract

All strings land in BOTH `en/` and `fr/` namespaces (EN/FR parity is a hard constraint). No editorial verbs anywhere — the project denylist (`recommend` / `should` / `poor` / `good` / `better` / `worse-as-judgement` / `at-risk-as-judgement` / high-med-low confidence words) is enforced; "at-risk" and "overdue" are permitted ONLY as neutral time-bucket labels, never as advice (D-00). Numbers never embedded pre-formatted in strings (use `fmt*` + i18n interpolation count). Em-dash (`—`) is the "not determinable" sentinel, verbatim with the shipped sentinel idiom.

| Element | Copy (EN — FR parity required) |
|---------|------|
| View / section heading | "OS end-of-support forecast" (factual noun phrase, no verb). |
| As-of reference line (D-07) | "Forecast computed as of {today}" — `{today}` = workbook-load date, locale-formatted via `fmt*`. Always visible. Factual, explains the wall-clock-coupled semantics without judgement. |
| Catalogue freshness line (D-03, always shown) | "EOS catalogue verified: {YYYY-MM-DD}" — `{date}` from the bundled catalogue `lastVerified` build constant, locale-formatted. Always visible, neutral. |
| Catalogue staleness caption (D-03, conditional >90 days) | "Catalogue may be outdated" — shown ONLY when `lastVerified` is >90 days before `{today}`; neutral caption, no verb, no warning glyph, no color (renders in the same neutral `text-slate-500`/dark `text-slate-400` as the freshness line). |
| Bucket labels (the 6 peer buckets) | Overdue: "Overdue". Time horizons: "Within 3 months" / "Within 6 months" / "Within 9 months" / "Within 12 months" (cumulative `≤ +N` scheme — see Open Item 2). Unknown OS: "Unknown OS". All are factual time/state labels; none implies advice. Discrete-vs-cumulative final wording reconciled in Open Item 2. |
| Bucket count | The figure only (mono, tabular). The accessible name is "{label}: {count} {entityKind}" (e.g. "Overdue: 12 VMs") for screen readers — entity kind explicit so host vs VM cardinality is never conflated (D-09b). |
| Reconciliation line (D-06/D-10) | "{total} entities — {sumOfBuckets} bucketed" rendered as a factual check line; total and sum are equal by construction (counts reconcile, nothing dropped). If a split by entity kind is shown: "VMs: {n} · ESXi hosts: {m}" — never a single conflated total (D-09b). |
| ESXi host classification surfacing (D-09a/b/c) | Host lifecycle state is a factual word/date derived from the catalogue; em-dash `—` sentinel when a level (patch/update) is not determinable from catalogue fields actually present (D-09c — never a fabricated level). Host and VM counts are presented as **distinct entity kinds**, never summed into one number (D-09b). If a unified timeline is chosen, the count is split by kind in the label. |
| Bucket → entity drill affordance | Each bucket tile is a `<button>` whose accessible name is "Show {label} entities ({count})". Clicking opens the shipped P3 `DataTable` drill (D-08) filtered to that bucket's entities. No new table component, no new verb — the affordance label is factual. |
| Unknown-OS drill content (D-11 — mandatory) | The unknown-OS drill MUST surface the **raw unrecognized OS strings preserved verbatim (D-12) with per-string occurrence counts**. Heading: "Unrecognized OS strings". Each row: the verbatim original string (mono) + "{count} occurrences" (mono, tabular). Aggregated-count-only is explicitly NOT acceptable — the raw strings are the maintainer signal and the user-transparency contract. |
| Empty state heading (no workbook loaded) | Owned by the existing app-shell upload/empty flow — unchanged. The EOS view is not reachable without a loaded snapshot. |
| Empty state — bucket has zero entities | The bucket tile still renders with count `0` (it is a real reconciled fact, not absence). No "no data" copy on a zero bucket — `0` is the truthful value. The drill for a zero bucket opens an empty `DataTable` with the table's own shipped empty presentation (no new copy). |
| Empty state — zero unknown-OS strings | Unknown-OS bucket shows `0`; its drill heading still renders with no rows (table's shipped empty state). Factual, no "all good" editorializing. |
| Not-determinable sentinel | Em-dash `—` for any lifecycle date/level not computable from parsed RVTools columns + the bundled catalogue (calc-from-real-data; never a fabricated date or `0`). |
| Error state | No new error surface. Catalogue is a build-time validated static import (Zod at the build boundary, D-01) — there is no runtime fetch and therefore no runtime catalogue-fetch error. Parse/load errors remain owned by the existing app-shell error boundary + `sonner` toast — unchanged. |
| Destructive confirmation | None — there are NO destructive actions in this phase. The forecast is a read-only derived projection; no inputs, no mutation, no confirmation dialogs. |

## Interaction Contract

- **Keyboard:** the (5-segment, if Open Item 1) `ViewToggle` keeps Tab-into-group + Arrow Left/Right/Up/Down move-and-activate with wraparound + Space/Enter activate (native `<button>`) verbatim. Each bucket tile is a native `<button>` (keyboard-operable by construction; `aria-label` = "Show {label} entities ({count})", EN/FR). The drill `DataTable` keeps its shipped keyboard behavior unchanged. Selected-bucket state is conveyed by both the neutral ring AND `aria-pressed` on the bucket button (do not rely on color alone — and there is no color used here anyway).
- **One-click drill (D-08):** clicking/activating a bucket tile opens the shipped `DataTable` filtered to that bucket's entities — one click, reuse existing table, no intermediate modal. Re-clicking the active bucket (or selecting another) swaps the drill content; the bucket strip stays visible (the user keeps the reconciled overview while drilling).
- **Unknown-OS transparency (D-11):** activating the unknown-OS bucket surfaces the verbatim raw unrecognized OS strings + occurrence counts (not an aggregated count). The original RVTools string is displayed unmangled (D-12 — case/whitespace normalization is matching-only, never display).
- **Reconciliation invariant (visible):** the sum of all bucket counts equals the entity total by construction; the reconciliation line makes this checkable on screen. Host vs VM cardinality is never conflated into one number (D-09b) — entity kind is explicit in counts, labels, and accessible names.
- **Single-memo invariant:** the entire EOS projection (per-bucket counts, affected-entity lists, raw unknown strings, `lastVerified`) flows through the existing `useEstateView` / `buildEstateView`. No new `useMemo`, no recompute in components (presenters only) — D-00.
- **Privacy / no-network:** the catalogue is a build-time static import validated by Zod at the build boundary (D-01). There is NO runtime fetch (the P1 guard throws by design). No `localStorage` of dataset rows or bucket selection; refresh = data gone. The selected-bucket state is ephemeral component `useState`.
- **Factual presentation (D-00 / G1):** no editorial verb, no verdict color, no status icon, no traffic-light. "Overdue"/"at-risk" are neutral time labels only. Em-dash sentinel for any not-determinable lifecycle level (D-09c) — never a fabricated date or zero.

## Claude's-Discretion Recommendations (within locked intent)

**Open Item 1 — View placement: 5th ViewToggle segment vs Dashboard section vs drill (D-05):**
RECOMMENDATION: **Add a 5th `'eos'` `ViewToggle` segment**, mirroring P5 'hosts' and P6 'planning' exactly. Rationale: the EOS forecast is a first-class top-level analytical lens with its own bucket strip, chart, and drill — peer to Dashboard/Inventory/Hosts/Planning, not a sub-section of any of them. The shipped `ViewToggle` extension path is proven twice (P5/P6), DRY, and keeps the nav model stable. A Dashboard section would crowd the dashboard and bury the reconciled buckets; a drill-only treatment would hide a first-class capability. Contract decision for the executor: extend `AppView` + `VIEWS` with `'eos'`, add `nav.eos` (EN/FR), branch the App/shell view-state exactly like the P6 'planning' branch. No new nav component.

**Open Item 2 — Bucket scheme: discrete non-overlapping vs cumulative `≤ +N` (D-06):**
RECOMMENDATION: **Cumulative `≤ +N` thresholds** ("Within 3 / 6 / 9 / 12 months"), plus the explicit **Overdue** bucket and the first-class **Unknown OS** bucket as peers. Rationale: cumulative answers the real user question ("how many entities lose support within the next N months") directly and scans intuitively (each later bucket ⊇ the earlier — monotonic, no mental subtraction). It maps cleanly to an ECharts calendar/heatmap timeline (cumulative density over the horizon reads naturally as a ramp). Hard constraint preserved: counts still **reconcile to the entity total** — reconciliation is computed on the *disjoint partition* (overdue ∪ disjoint-window ∪ beyond-12mo ∪ unknown-OS = total), and the cumulative tile counts are a presentation overlay on top of that partition; the reconciliation line is computed from the disjoint partition so nothing is double-counted or dropped. Executor must implement: engine produces the disjoint partition (the reconcilable truth); the bucket strip displays cumulative `≤+N` tiles derived from it; the reconciliation line uses the partition, not the cumulative sum. If the engine/chart fit favors discrete windows at plan time, discrete is an acceptable fallback within D-06 — but cumulative is the recommended default. Labels factual either way.

**Open Item 3 — Unknown-OS raw-string presentation: peer-drill list vs adjacent diagnostics affordance (D-11):**
RECOMMENDATION: **Render the verbatim raw-string + occurrence-count list AS the content of the unknown-OS bucket drill** (not a separate hidden diagnostics panel). Rationale: the unknown-OS bucket is a first-class peer (D-10); its drill is the natural place a user (and the maintainer) looks. Making the raw strings the drill body — instead of routing the unknown bucket through the entity `DataTable` like the other buckets — satisfies D-11 directly (raw strings + counts, never aggregated-only), keeps it discoverable in one click (D-08 affordance parity), and avoids a hidden diagnostics mode the maintainer might miss. The other five buckets drill into the shipped entity `DataTable`; the unknown-OS bucket drills into this raw-string list (it is a different question — "which strings did we fail to recognize" vs "which entities are in this window"). Both are one click, both factual. Contract decision for the executor: unknown-OS bucket → raw-string/occurrence list (mono, verbatim, D-12); the other buckets → shipped `DataTable`.
