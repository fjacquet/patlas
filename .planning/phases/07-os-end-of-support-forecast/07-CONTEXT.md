# Phase 7: OS End-of-Support Forecast - Context

**Gathered:** 2026-05-17
**Status:** Ready for planning

<domain>
## Phase Boundary

Deliver the OS/ESXi lifecycle-forecast capability: a **bundled, Zod-validated** `endoflife.date` catalogue (baked at build time — the P1 privacy guard throws on any runtime non-same-origin fetch, so there is **zero** live API call), an OS-string normalizer (regex bank handling RHEL 8's 4 variants and Oracle Linux's 3, case-insensitive), an ESXi build→support-state classifier, lifecycle bucketing (time horizons + explicit **overdue** + first-class **unknown-OS** buckets), one-click bucket→affected-entity drill, and a `lastVerified` catalogue date surfaced factually with a CI staleness warning at >90 days.

**Requirements owned:** EOS-01..06.

**Explicitly OUT of P7:**
- Hardware vendor EOS (never promised — not in RVTools; P5 D-08).
- Any runtime network call (catalogue is build-time only — privacy invariant).
- The HTML/PPTX export of the EOS view (→ P10; P7 makes the view, not the export).
- Trends/multi-snapshot evolution of EOS (→ P8).
- Regressing P5's plain-text ESXi/host presentation.
</domain>

<decisions>
## Implementation Decisions

### Cross-cutting (carried — locked, not re-asked)
- **D-00:** Calc-from-real-data (P4 D-01) + G1 factual-presentation lesson: buckets are **date facts, not advice** — no editorial verbs (no "should upgrade / poor / at-risk-as-judgement"); "at-risk"/"overdue" are neutral time-bucket labels only. Privacy/in-memory invariant, single `useMemo` (`useEstateView`) — the EOS projection composes inside `buildEstateView`, NO new memo — EN/FR parity, no `localStorage` of dataset rows, em-dash sentinel for "not determinable" all hold. Every value derived only from parsed RVTools columns + the bundled catalogue; never fabricated.

### Catalogue sync & freshness (Area 1)
- **D-01:** The `endoflife.date` catalogue is **bundled and Zod-validated**, with **zero runtime fetch** (privacy guard throws by design). The exact bundling mechanism — committed JSON snapshot regenerated by an explicit `sync:eos`-style script vs a CI build-step fetch — is **Claude's discretion** for researcher/planner, decided on the determinism-vs-freshness tradeoff. Hard constraint either way: no network at app runtime; the catalogue is a static import validated by Zod at the build/validation boundary (engines stay pure — Zod lives only at this boundary, mirroring the parser-boundary rule).
- **D-02:** If `endoflife.date` is unreachable when the catalogue is refreshed/built: **use the last good snapshot and surface staleness** — a third-party outage MUST NEVER block a vatlas deploy. CI **warns** the maintainer when the snapshot is >90 days old; it never blocks the deploy. (A refresh-script may exit non-zero to prevent committing a bad snapshot, but the normal validate-only CI deploy still succeeds — exact failure semantics are planner's call within "outage never blocks deploy".)
- **D-03:** Freshness UX: the EOS view **always** shows a neutral factual line "EOS catalogue verified: YYYY-MM-DD"; when the catalogue is >90 days old it additionally shows a neutral caption ("catalogue may be outdated" — factual, no editorial verb), EN/FR parity. CI separately warns the maintainer (D-02).
- **D-04:** **Single EOL date model** — use endoflife.date's primary standard-support EOL date only; **do not** model paid extended-support tiers (RHEL ELS / Windows ESU). An OS past standard EOL but under paid ESU still buckets as **overdue** (factually true: standard support ended). Simpler and deterministic.

### View placement, bucket semantics & drill (Area 2)
- **D-05:** View placement (new 5th ViewToggle segment vs Dashboard section vs drill) is **Claude's discretion** — if a segment, reuse the shipped `fieldset`+`aria-pressed` `ViewToggle` idiom (precedent: P5 added Hosts, P6 added Planning); do NOT build a new nav component.
- **D-06:** Bucket definition (discrete non-overlapping windows vs cumulative ≤+N thresholds) is **Claude's discretion**, within these hard constraints: must include an explicit **overdue** bucket and a **first-class unknown-OS** bucket (D-09); counts must **reconcile to the entity total** (nothing silently dropped); whichever scheme maps cleanest to the drill + the Phase-2 ECharts chart (calendar/heatmap available). Factual labels only.
- **D-07:** **Reference date for bucketing = TODAY (the date the user loads the workbook), not the snapshot capture date.** "Overdue" = EOL strictly before today. Surface, factually, that the forecast is computed "as of {today}". *User-chosen tradeoff explicitly accepted:* the same workbook yields different buckets on different days (non-reproducible, wall-clock-coupled) — this is the user's deliberate decision; do not silently substitute capture-date. (Catalogue `lastVerified` from D-03 is a separate date and is also surfaced.)
- **D-08:** Drill = **reuse the shipped P3 inventory `DataTable` + drill idiom** (sortable/filterable, same component family) for bucket→affected-entity list — no new table component (DRY). Exact wiring is planner/UI-spec's call within "one click, reuse existing table".

### ESXi host EOS scope (Area 3)
- **D-09a:** ESXi build→support-state classification placement (EOS view only vs also augmenting the P5 Hosts view's ESXi column) is **Claude's discretion**, within: P7 owns the classification; **do NOT regress P5's plain-text Hosts view** (if augmented, regression-gate it); factual, no verdict.
- **D-09b:** Whether ESX hosts share one unified timeline with VM OSes or sit in a separate ESXi-lifecycle section is **Claude's discretion**, with the hard constraint: **never conflate host vs VM cardinality into a single misleading total** (host count and VM count are different entities — if unified, counts must be split by type).
- **D-09c:** ESXi granularity = patch + major per EOS-04, but **derive only from catalogue fields actually present** — classify against major-version EOL and, where the catalogue exposes it, patch/update-level EOL; em-dash sentinel when a level isn't determinable; never fabricate a level. Researcher confirms endoflife.date ESXi granularity; planner finalizes.

### Unknown-OS handling & feedback (Area 4)
- **D-10:** The **unknown-OS bucket is first-class** — a peer bucket beside overdue / time-horizon buckets, with its own count and drill. Counts always reconcile to the entity total; unmatched OSes are **never silently dropped** (EOS-05 intent).
- **D-11:** The unknown-OS drill MUST surface the **actual raw unrecognized OS strings with occurrence counts** somewhere (transparency for the user AND a concrete signal for the maintainer to extend the regex bank — ROADMAP intent). Exact presentation (peer-drill list vs adjacent diagnostics affordance) is **Claude's discretion**; aggregated-count-only is NOT acceptable.
- **D-12:** **Minor-4 locked:** OS-string matching is **case-insensitive and whitespace-normalized**, but the **original RVTools OS string is preserved verbatim** for display in drills and the unknown-OS list (never mangle the displayed original).

### Claude's Discretion (summary)
Bundling mechanism (D-01); exact build-failure semantics within outage-never-blocks (D-02); view placement segment-vs-section (D-05); bucket scheme discrete-vs-cumulative (D-06); drill wiring (D-08); ESXi placement & unified-vs-separate & granularity finalization (D-09a/b/c); unknown-OS presentation (D-11). All within the locked factual/privacy/reconcile constraints above. Exact label strings: factual, EN/FR parity, no editorial-verb denylist.
</decisions>
