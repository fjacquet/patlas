<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| endoflife.date → `scripts/sync-eos.mjs` (build time only) | Untrusted third-party JSON crosses here at maintainer-run sync time; never at app runtime |
| `catalogue.json` → `catalogue.ts` (Zod boundary) | Committed data file parsed/validated before any engine consumes it |
| CI workflow → committed snapshot | Read-only; no network egress in the deploy job |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-01 | Tampering | `catalogue.json` poisoned/malformed snapshot | mitigate | `EosCatalogueSchema.parse` at the `catalogue.ts` boundary throws on bad shape (build fails on a bad commit); JSON is PR-reviewable diff; single trusted HTTPS source in `sync-eos.mjs` |
| T-07-02 | Information disclosure | "Freshness fetch" leaking workbook data at runtime | mitigate | Architecturally impossible: D-01 mandates zero runtime network; P1 privacy guard throws on non-same-origin fetch; CI freshness step contains no `curl`/`fetch` (grep-gated == 0) |
| T-07-03 | Denial of service | Third-party outage blocking the GitHub Pages deploy | mitigate | CI deploy never fetches; `sync:eos` (maintainer-run, off the deploy path) exits non-zero without writing; freshness step is `exit 0` warn-only (D-02) |
| T-07-04 | Tampering | Malicious code (`eval`/`Function`) in catalogue data | accept | Catalogue is parsed as inert JSON data only; no `eval`/`Function`; strings are never executed — low risk, no mitigation beyond Zod typing |
</threat_model>
