<domain>
## Phase Boundary

Ship the two synthesis surfaces that close the loop ("the report is the product"): a single-file self-contained HTML report and a factual PPTX deck, both consuming the shipped `EstateView` shape, plus the i18n/theme/freshness polish and the GitHub Pages deploy. Architecture is already locked upstream (CLAUDE.md tech-stack decision record + ROADMAP "Pitfalls owned"): Live-DOM/`renderToStaticMarkup` + inline SVG for HTML, pptxgenjs (vsizer reuse) for PPTX, base64 self-hosted fonts, `<5 MB` / `<15 MB` size budgets, Web-Worker synthesis, Midnight Executive palette, factual-only denylist, GitHub Pages CI shape. This phase clarifies the **product choices** layered on that locked architecture; it does NOT re-decide the architecture, and it adds no new analytics capability (every number comes from existing view-models).

<decisions>
## Implementation Decisions

### PPTX deck shape
- **D-01:** One slide per cluster, **always** (faithful 1:1, the vsizer behavior) — no top-N cap / rollup slide even for large (50+ cluster) estates. Accept long decks for large estates.
- **D-02:** The PPTX is a **tighter executive subset**, not a mirror of the HTML report — low text density, headline numbers + charts per the fixed PPT-03 slide list (title · overview · per-cluster · CPU-Ready annex conditional · EOS · DR · trends · inventory summary). Exhaustive detail/annex lives in the HTML report only.
- **D-03:** Title slide carries **factual estate identity + freshness**: vCenter name(s), capture date, VM/host/cluster counts, "Generated by vatlas — factual estate synthesis". Brand-free Midnight Executive (PPT-02): no client logo, no editorial tagline.

### Export trigger UX
- **D-04:** **Two separate buttons in the existing app header** — "Export HTML" and "Export PPTX" — beside the Language/Theme toggles. No new ViewToggle segment, no new route/view; reuse the shipped header-control idiom.
- **D-05:** Downloaded filename convention: `vatlas_{vCenter}_{captureDate}.{ext}` (e.g. `vatlas_spvspherevc11_2026-05-18.html` / `.pptx`). vCenter token sanitized (filesystem-safe); ISO capture date; multi-vCenter → primary vCenter token or `multi`.
- **D-06 (Claude's discretion — see below):** progress feedback during the 5–30 MB Web-Worker synthesis.

### Locale & multi-snapshot scope of exports
- **D-07:** Exported HTML/PPTX carry the **active locale only** (whatever FR/EN the app is set to at export time) — one artifact = one language; numbers locale-formatted per PPT-04 (FR U+202F→U+00A0 substitution, EN `,`). No in-file FR/EN toggle, no dual-file output.
- **D-08:** Export body describes the **active snapshot's estate** — deliberately NOT the multi-vCenter merged estate the dashboard shows when several snapshots are selected — PLUS a trends section computed across all loaded snapshots. **Planner must honor this divergence:** the export consumes the *active snapshot's* `EstateView`, not the merged P4 view.
- **D-09:** With only **one snapshot loaded**, omit the trends section entirely (no HTML trends section, no PPTX trends slide) — factual: nothing to trend with a single snapshot.

### Claude's Discretion
- **D-06:** Worker progress feedback — user said "you decide". Documented choice: reuse the already-mounted `sonner` `<Toaster/>` in `App.tsx` — "Generating report…" toast → success ("Report ready") / error toast; the triggering export button is disabled with a spinner while the worker runs (UI stays interactive, synthesis off the main thread). Lightest adequate pattern, no new UI surface. Planner may refine but must not add a modal/blocking overlay.
- **HTML report IA & depth:** user did NOT select this area for discussion → follows ROADMAP's fixed HTM-04 section list (cover · executive headlines · per-cluster · EOS forecast · DR results · trends · annex · methodology footer) under the `<5 MB` typical / `<15 MB` hard-ceiling budget. Claude's discretion *within* those constraints: per-cluster depth vs budget (e.g. all clusters inline vs top-N inline + remainder in annex), and whether P9 Storage/Network get dedicated sections or fold into per-cluster/annex. Researcher should size this against the budget on a realistic large estate.

</decisions>

## Specific Ideas

- PPTX is faithful 1:1 per cluster even at 50+ clusters (D-01) — user explicitly rejected a top-N cap; long decks for large estates are acceptable.
- Export is "what you see is what you export" for locale (D-07) and snapshot (D-08): the active snapshot, the active language — NOT the merged multi-vCenter estate. This is a deliberate, planner-binding divergence from the dashboard's merged view.
- Filename must be self-describing and filesystem-safe: `vatlas_{vCenter}_{captureDate}.{ext}` (D-05).
