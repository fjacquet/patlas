# Phase 10 — UI Design Contract

> Visual and interaction contract for **HTML + PPTX Exports & Deploy**. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Scope split (binding).** This phase's UI surface is *small in the app* and *large in the exported artifact*:
> 1. **In-app:** two header buttons + a progress toast + disabled-button spinner. REUSES the shipped Midnight Executive system and the shipped header-control idiom. Introduces NO new in-app design mechanism.
> 2. **Exported HTML report:** IS the product. The real visual contract — self-contained, offline, JS-disabled-safe document styling.
> 3. **PPTX deck:** governed by pptxgenjs layout (not CSS). Captures only the palette→fill mapping + text-density + factual-content contract the slide builder must honor.
>
> Every design decision below is pre-populated from `10-CONTEXT.md` (D-01..D-09), `10-RESEARCH.md`, `ROADMAP.md` § Phase 10, `REQUIREMENTS.md` (HTM/PPT/DEP), and the shipped codebase. No open questions for the user — the architecture is locked upstream and the design system is shipped (8 phases).

## Spacing Scale

The shipped system uses Tailwind v4 defaults (4px base unit — already a strict multiple-of-4 scale). This phase adds **no new spacing tokens**. The two new in-app affordances reuse exactly the shipped header-control spacing.

| Token | Value | Usage in this phase |
|-------|-------|---------------------|
| xs | 4px | `gap-1` between glyph and label inside an export button; `p-0.5` fieldset-style inner pad (matches `ThemeToggle`/`LanguageToggle`) |
| sm | 8px | `gap-2` between the export buttons and the existing Language/Theme toggles in the header control cluster (matches `App.tsx` header `gap-2`) |
| md | 16px | not introduced by this phase in-app (header is compact); HTML report body block rhythm |
| lg | 24px | HTML report intra-section padding |
| xl | 32px | HTML report section separation (matches the shipped `ClusterDetail` `p-8` screen-fit precedent that doubles as the PPTX slide bound) |
| 2xl | 48px | HTML report major section breaks (cover → headlines → per-cluster …) |
| 3xl | 64px | HTML report cover vertical centering |

**In-app button metrics (locked, mirrors shipped header buttons):** `rounded px-2 py-1 text-xs` inside a `rounded-md border ... p-0.5` group container — pixel-identical to `ThemeToggle`/`LanguageToggle` so the two export buttons read as the same control family. No new size.

**HTML report spacing (document, not app):** an 8px-multiple vertical rhythm — section padding `lg` (24px), section gap `2xl` (48px), inline label/value gap `sm` (8px). The report is a *document*, so generous whitespace is correct; it is NOT the app's compact density.

**PPTX spacing:** governed by pptxgenjs inches/EMU coordinates in the ported `builder.ts`/slide modules — NOT this scale. The screen-fit `ClusterDetail` layout (one viewport) is the per-cluster slide bound (D-01, one slide per cluster always).

Exceptions: none introduced. (PPTX uses pptxgenjs's own coordinate system by necessity — out of scope for an 8px-grid assertion.)

## Typography

The app uses Tailwind text utilities over the shipped font stack. This phase adds **no new app type tokens**. Two declared roles cover the in-app surface; the HTML report declares a *document* scale (a report is read, not operated).

### In-app (header buttons + toast) — REUSES shipped idiom

| Role | Size | Weight | Line Height | Source / usage |
|------|------|--------|-------------|----------------|
| Control label | 12px (`text-xs`) | 600 (`font-semibold` on active, 500 otherwise) | 1.5 (Tailwind default) | Export button label — identical to `ThemeToggle`/`LanguageToggle` button text |
| Toast text | 14px (sonner default `text-sm`) | 400 | 1.5 | sonner `<Toaster/>` default; no override (D-06) |

Exactly **2 in-app roles, 2 weights** (regular 400 / semibold 600) — within the contract ceiling and matched to shipped controls.

### HTML report (document scale) — exported artifact only

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Body | 14px | 400 | 1.5 | Table cells, annex prose, methodology footer |
| Label | 12px | 600 | 1.4 | KPI captions, axis-adjacent labels, section eyebrows |
| Heading | 20px | 600 | 1.25 | Section headings (per-cluster, EOS, DR, trends, annex) |
| Display | 28px | 600 | 1.2 | Cover title + executive-headline KPI figures |

4 document roles, 2 weights (400 / 600). Numerics that compare across rows use the mono stack with `tabular-nums` (shipped precedent: `ClusterDetail` `font-mono ... tabular-nums`). Inlined into the report's single `<style>` block as plain CSS (no Tailwind runtime in a static file).

### PPTX

Type is set by the ported vsizer `theme.ts`/slide modules (pptxgenjs `fontSize` in points), NOT CSS. Contract: **low text density, headline numbers + charts** (D-02 — "tighter executive subset, not a mirror of the HTML report"). Exhaustive prose lives in the HTML report only.

## Color

**Single source of truth: the shipped `src/index.css` `@theme` block (Midnight Executive).** This phase adds **zero new palette tokens**. Both exports inherit this palette.

> **Binding chart-color constraint (carried from Phase 9, do not regress):** zrender (the renderer ECharts bundles) cannot parse `oklch()` — an `oklch()` string silently falls back to `#000000`. Every chart-bound color MUST be the **sRGB hex** equivalent already resolved in `src/theme/echartsTheme.ts`. The exported HTML report (inline ECharts SSR SVG) and the PPTX deck (rasterized PNG) BOTH render through that same theme, so they inherit the sRGB-hex values automatically — **the export path must reuse `echartsTheme.ts`, never re-derive colors and never pass `oklch()` into a chart option.**

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | App: `bg-slate-50` / dark `bg-surface-900` (`oklch(15% 0.02 260)`). Report: `#ffffff` page, near-black `#0f172a` text | Page/surface background; report paper |
| Secondary (30%) | App: `.panel` `bg-white` / dark `bg-surface-800`, `border-slate-200` / dark `border-surface-700`. Report: `#f1f5f9` section bands, `#e2e8f0` rules | Cards, header bar, section bands, table zebra |
| Accent (10%) | Navy primary ramp — `--color-primary-500` `oklch(45% 0.18 270)` (chart sRGB hex `#3245b7`); active control `bg-primary-100 text-primary-900` (light) / `bg-primary-700 text-slate-100` (dark) | Chart series primary, active control state, in-report links/anchors, KPI emphasis |
| Gold marker | `--color-accent-500` `oklch(78% 0.16 75)` (≈`#F9B935`) | **Factual threshold/flag marker ONLY** — `bg-accent-500/15` + `border-l-2 border-accent-500` (shipped P9 idiom). Never an editorial flourish, never a CTA color. |
| Utilization (status, not verdict) | `--color-util-low` `#4aa342` / `--color-util-mid` `#ef8700` / `--color-util-high` `#df202e` (sRGB hex from `echartsTheme.ts`) | Gauge/utilization bands in inlined report charts ONLY — status, never a judgment. No traffic-light in tables/markers. |
| Destructive | **none** | This phase has **no destructive action** (export + download is non-destructive, idempotent, no data mutation). No destructive color is declared or used. |

**Accent reserved for:** (1) primary chart series (navy ramp via `echartsTheme.ts`); (2) the active state of the two export buttons (mirrors shipped Theme/Language active state exactly); (3) in-report anchor links and KPI figure emphasis. **Accent is NOT used for** general button fills, borders, the toast, or any "all interactive elements" sweep. The gold `--color-accent-500` remains exclusively the factual threshold marker (shipped P9 contract) — exports must carry that marker into the report's storage/threshold rows with the identical `bg-accent-500/15` + `border-l-2` treatment, no traffic-light.

**Static-file light/dark handling (HTML report, D-07 + FND-04 context):** The report is a *single static file with no JS*. It MUST NOT depend on the app's class-strategy `.dark` runtime toggle (the FOUC script does not ship in the report). Contract: **the exported report is light-theme, fixed.** A static document is read/printed; a runtime theme switch needs JS the report deliberately omits (HTM-02). The Midnight Executive *palette* (navy ramp, gold marker, util colors) is identical to the app's; only the surface is pinned light for print/offline fidelity. (`@media (prefers-color-scheme: dark)` is explicitly NOT used — it would make the same file render differently on different machines, breaking shareable-artifact predictability and print output.)

**PPTX color:** the ported vsizer `theme.ts` Midnight Executive hex constants (no `#` prefix per pptxgenjs convention) map to slide fills/text. Brand-free (PPT-02 / D-03): no client logo, no editorial color. Charts are rasterized PNG (PowerPoint cannot render SVG — Research Pitfall 1) so they carry the `echartsTheme.ts` sRGB palette baked into the bitmap.

## Copywriting Contract

**Factual-only invariant (HTM-05 / PPT criterion 4 / CLAUDE.md i18n rule), binding for every string below:** no `recommend`, `should`, `poor`, `good`, `critical`, `warning`, no traffic-light language, no editorial verb. No pre-formatted numbers inside any string — all numbers interpolated via `@/utils/format` with the active locale (D-07; Moderate-2). EN+FR parity mandatory (Minor-7 key-diff CI gate). All new keys land in the **net-new `report` namespace** (HTML) and a **net-new `pptx` namespace** (deck) — both must be added to `NAMESPACES` in `src/i18n/index.ts` and to `locales/{en,fr}/`.

> **Doc-comment denylist trap (CLAUDE.md Gotchas / Research anti-pattern):** the grep gate + security hook match comments too. Any code comment describing the *absence* of a forbidden token must NOT spell the literal token. Phrase "no editorial verbs here" without naming one.

| Element | Copy (EN) | Copy (FR) |
|---------|-----------|-----------|
| Primary CTA — HTML | **Export HTML** | **Exporter HTML** |
| Primary CTA — PPTX | **Export PPTX** | **Exporter PPTX** |
| Button ARIA label — HTML | `Export the estate report as a self-contained HTML file` | `Exporter le rapport du parc en fichier HTML autonome` |
| Button ARIA label — PPTX | `Export the estate report as a PPTX deck` | `Exporter le rapport du parc en présentation PPTX` |
| In-progress toast | `Generating report…` | `Génération du rapport…` |
| Success toast | `Report ready — {filename}` | `Rapport prêt — {filename}` |

| Disabled-button state (during synthesis) | label unchanged + inline spinner glyph + `aria-busy="true"` + `disabled` | (same; label = FR label) |

| HTML report — single-snapshot trends omission (D-09) | Section omitted entirely (no placeholder, no "no data" line) — factual: a single snapshot has nothing to trend | (same — section absent) |

## Interaction & State Contract (in-app, this phase's new behavior)

| Surface | Contract |
|---------|----------|
| Button placement | In the `App.tsx` header control cluster `<div className="flex items-center gap-2">`, **after** `<ViewToggle/>` and **before** `<LanguageToggle/>` (or grouped immediately before Language/Theme). Order: ViewToggle · **Export HTML · Export PPTX** · Language · Theme. No new ViewToggle segment, no route (D-04). |
| Visual treatment | Two `<button type="button">` elements in one `<fieldset>`-style group container reusing the EXACT classes from `ThemeToggle`/`LanguageToggle` (`rounded-md border border-slate-200 bg-white p-0.5 ... dark:border-surface-700 dark:bg-surface-900`, inner buttons `rounded px-2 py-1 text-xs`). They read as one more control of the same family. Each button: a 14×14 inline stroke-SVG glyph (download/document idiom, authored like `ThemeToggle`'s glyphs) + the text label. |
| Idle → busy | On click: triggering button → `disabled`, `aria-busy="true"`, glyph swapped for an inline spinner SVG (CSS `animate-spin`); the OTHER export button also disabled (one synthesis at a time). sonner in-progress toast shown. The rest of the app stays fully interactive (synthesis is in the Web Worker). |
| Busy → done | Worker posts result → Blob → `URL.createObjectURL` → `<a download>` click → revoke. In-progress toast replaced by success toast (`Report ready — {filename}`). Both buttons re-enabled, `aria-busy` cleared, spinner → original glyph. |
| Busy → error | Worker posts error → error toast (factual copy above). Buttons re-enabled. No data mutation occurred; nothing to roll back. NO modal, NO blocking overlay (D-06 hard constraint). |
| Focus / a11y | Buttons are real `<button>`s, keyboard-activable, visible focus ring (browser default + the shipped controls' `transition-colors`; do not remove outline). `aria-busy` on the active button during synthesis; `aria-label` per the copy table (verb + what is produced). The group container carries an `aria-label` legend (sr-only `<legend>`) exactly like `ThemeToggle`/`LanguageToggle`. Toast region is sonner's default polite live region (already mounted) — no extra ARIA needed. |
| Disabled affordance | Native `disabled` + reduced-opacity is acceptable (no custom disabled token introduced); the spinner is the primary "work in progress" signal, the toast is the secondary. |

## HTML Report Visual Contract (the exported product)

| Concern | Contract |
|---------|----------|
| Self-containment | One `<!doctype html>` string. Inline `<style>` only (no external CSS). Fonts: system-font stack by default (0 bytes); a base64 `@font-face` woff2 ONLY if it fits the budget (default = no embedded font). NO `<link>`, NO `<script>`, NO network reference of any kind (the privacy guard throws on any fetch — Research Pitfall 3). |
| CSP meta | `<meta http-equiv="Content-Security-Policy" content="default-src 'none'; img-src data:; style-src 'unsafe-inline' data:; font-src data:">` — the artifact executes no script (HTM-02). |
| Charts | Raw inline `<svg>…</svg>` from ECharts SSR (`renderToSVGString()`), NOT `<img>`. Crisp at every zoom, no JS to view (HTM-02). Colors via `echartsTheme.ts` sRGB hex (the oklch constraint). |
| Sections (HTM-04, fixed order) | cover · executive headlines · per-cluster · EOS forecast · DR results · trends *(omitted when <2 snapshots — D-09)* · annex · methodology footer. |
| Per-cluster depth vs budget (Claude's discretion within HTM-04 / Research Open Q3) | **Top-N clusters inline with charts; remainder summarized as a chart-less annex table.** Research-recommended N ≈ **12–20** clusters inline. P9 Storage/Network fold into the annex (table form), not dedicated per-item chart sections. This is the size lever — the planner picks the exact N and MUST add a byte-length assertion test on the `scripts/generate-inventory-10k.mjs` fixture. |
| Size budget (HTM-03 / Moderate-7) | **< 5 MB typical** (≈5000-VM estate), **< 15 MB hard ceiling** (≈36000-VM / 60+ cluster). Enforced by a size-accounting assertion in `inlineAssets.ts` + a fixture test. Font bytes default to 0 (system stack) to protect the 5 MB target. |
| Anchor ids (Research Pitfall 6) | Deterministic, namespaced: `c-${index}-${slug(name)}` — never raw user cluster/datastore names (real data repeats names). One snapshot per report (D-08) so no snapshot prefix, but section+index prefixing is mandatory to avoid duplicate-id link breakage. |
| XSS (Security Domain) | All user-derived strings (VM / datastore / portgroup / cluster names) go through normal React JSX text nodes in `renderToStaticMarkup` (auto-escaped). The raw-HTML-injection React prop is not used for any data. Chart SVG comes ONLY from ECharts SSR (a trusted generator), never assembled from user strings. |
| Theme | Fixed light (see Color §). No JS theme toggle, no `prefers-color-scheme` branch — predictable shareable/printable artifact. |
| Scope (D-08) | Body = the **active snapshot's** `buildEstateView`, NOT the multi-vCenter merged dashboard view. Trends section = `buildEstateView` over ALL loaded snapshots (≥2 only). This divergence is planner-binding. |

## PPTX Deck Visual Contract

| Concern | Contract |
|---------|----------|
| Palette | Ported vsizer `theme.ts` Midnight Executive hex (no `#` prefix). Brand-free: no logo, no editorial tagline (PPT-02 / D-03). |
| Charts | **Rasterized to PNG** (`primitives/chartSvg.ts`) — PowerPoint clients do NOT render embedded SVG (Research Pitfall 1, verified). The PNG bakes in the `echartsTheme.ts` sRGB palette. Never `addImage` an SVG data URI. |
| Slide list (PPT-03, fixed) | title · overview · per-cluster · CPU-Ready annex *(conditional)* · EOS · DR · trends *(omit if <2 snapshots — D-09)* · inventory summary. |
| Per-cluster (D-01) | **One slide per cluster, always.** No top-N cap, no rollup slide, even at 50+ clusters. Long decks for large estates are accepted (explicit user decision). The shipped screen-fit `ClusterDetail` layout is the per-cluster slide bound. |
| Text density (D-02) | Tighter executive subset — headline numbers + charts, low prose. Exhaustive detail lives ONLY in the HTML report. |
| Title slide (D-03) | Factual estate identity + freshness: vCenter name(s), capture date, cluster/host/VM counts, `Generated by vatlas — factual estate synthesis`. Numbers via locale-parameterized `format.ts`. |
| Numbers (PPT-04 / Moderate-2 / Research Pitfall 4) | Locale-formatted via the **locale-parameterized** ported `format.ts` (vsizer's was FR-only — must accept a locale arg). FR: `,` decimal + U+202F→U+00A0 thousands substitution. EN: `.` decimal + `,` thousands. Active locale only (D-07), posted into the worker. Golden-PPTX snapshot test asserts both locales. |
| Scope (D-08) | Same active-snapshot rule as the HTML report; trends slide from all snapshots (≥2). |
