# Phase 2 — UI Design Contract

> Visual and interaction contract for the Aggregation & Global Dashboard phase. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
> Scope: single-snapshot global dashboard — global summary card, one-column-per-cluster grid, OS-family donut, CPU%/RAM%/allocation gauges, CPU Ready panel, three-accounting-mode toggle. All charts via the Phase-2 `<Chart>` wrapper (ECharts, SVG renderer only). Coexists with the existing Phase-1 `SnapshotListSidebar`.

## Spacing Scale

Declared values (multiples of 4; Tailwind default scale — no custom scale invented, per `<must_specify>` "reuse vsizer/Tailwind tokens"):

| Token | Value | Tailwind | Usage |
|-------|-------|----------|-------|
| xs | 4px | `p-1` `gap-1` | Icon gaps, inline label-to-value gap inside a stat |
| sm | 8px | `p-2` `gap-2` | Compact stat-row spacing, chip padding |
| md | 16px | `p-4` `gap-4` | Default — `.panel` padding (shipped), card internal padding, grid gap between cluster columns |
| lg | 24px | `p-6` `gap-6` | Dashboard section vertical rhythm (summary card → cluster grid → CPU Ready) |
| xl | 32px | `px-8` | Dashboard outer horizontal padding on wide viewports |
| 2xl | 48px | `gap-12` | Major break between dashboard regions when stacked on narrow desktop |
| 3xl | 64px | — | Reserved; not used Phase 2 |

Exceptions:
- Accounting-mode toggle and any keyboard-operable control: minimum 32px (`h-8`) hit height, ≥40px (`h-10`) recommended for the segmented toggle so the keyboard focus ring is unambiguous (desktop-first; no 44px touch target needed — mobile is explicitly out of scope per FEATURES.md A10).
- Per-cluster column minimum width 200px (`min-w-[200px]`) so figures + gauges stay legible; columns scroll horizontally past the viewport rather than shrinking below this (RVTools-admin grid expectation; not a spacing-scale violation — it is a min-width floor, not a spacing token).

## Typography

UI font: `system-ui` stack (shipped). Numeric/tabular font: `--font-mono`. Two weights only: regular 400 + semibold 600 (no other weights — KISS).

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Display | 28px (`text-3xl`) | 600 | 1.2 | Global summary headline number context only if a single hero metric is shown; otherwise unused — dashboard leads with the grid, not a hero number |
| Heading | 20px (`text-xl`) | 600 | 1.2 | Section titles ("Estate summary", "Clusters", "CPU Ready"), cluster column header (cluster name) |
| Body | 16px (`text-base`) | 400 | 1.5 | Metric values in summary card, gauge captions, donut legend labels |
| Label | 14px (`text-sm`) | 500 | 1.4 | Stat labels ("ESX", "VMs", "Physical GHz"), accounting-mode toggle option labels, units suffix; reuses shipped `.label` class (`text-sm font-medium`) |
| Numeric (tabular) | 16px (`text-base`) | 600 | 1.4 | Per-cluster figures (counts, GHz, %, ratio) — set in `--font-mono`, `tabular-nums`, right-aligned within a column so digits align vertically across clusters |
| Micro | 12px (`text-xs`) | 400 | 1.4 | "not reported" / accounting-mode echo ("Active") / capture-date provenance line / `—` sentinel caption |

Notes:
- Exactly 4 primary sizes in active use Phase 2: 14 / 16 / 20 / 28. The 12px micro size is a constrained exception reserved for provenance, sentinels, and the mode echo only (factual metadata, never primary content) — documented here so the checker treats it as an allowed 5th rather than a scale violation.
- Heading line-height 1.2; body/label line-height 1.4–1.5. No italic. No letter-spacing overrides.
- All numbers rendered through the ported `utils/format.ts` locale-aware formatters (see Number & Locale Formatting). Never hard-code a font-size on a number outside the Body/Numeric rows above.

## Color

The 60/30/10 split is expressed against the SHIPPED Midnight Executive tokens. Light and dark both mandatory; values below give the light value with the dark counterpart in the Usage column.

| Role | Token / Value | Usage |
|------|---------------|-------|
| Dominant (~60%) | `bg-slate-50` (light) / `bg-surface-900` (dark) | App background, dashboard canvas, gauge tracks-as-negative-space |
| Secondary (~30%) | `.panel` = `bg-white` border `slate-200` (light) / `bg-surface-800` border `surface-700` (dark) | Summary card, each per-cluster column card, CPU Ready panel, donut container, sidebar (shipped) |
| Accent (~10%) | `--color-accent-500` gold (`oklch(78% 0.16 75)` ≈ `#F9B935`) | RESERVED — see explicit list below |
| Primary brand | `--color-primary-*` navy scale | Active accounting-mode toggle segment background (`primary-600`), section heading text on light, focus ring (`ring-primary-500`), donut "Windows" series anchor |
| Status — low/mid/high | `--color-util-low` green / `--color-util-mid` orange / `--color-util-high` red | CPU%/RAM% gauge fill bands and the per-cluster utilization mini-bar fill ONLY; factual status thresholds, never an editorial verdict (see CPU Ready & status semantics) |
| Destructive | `--color-util-high` red | NOT used Phase 2 — no destructive actions exist in this phase (no delete/clear on the dashboard; snapshot removal lives in the Phase-1 sidebar `SnapshotCard`, out of this contract). Listed for completeness only. |

Accent (gold `--color-accent-500`) reserved for — explicit closed list:
- The "count of VMs above the 5% CPU Ready warning threshold" figure in the CPU Ready panel (DSH-05) — a single factual highlight number, never a label or background.
- The active-state underline/marker on the currently-selected accounting mode in the toggle, IF the navy primary fill is not used (pick exactly one active-state treatment in implementation — primary-fill OR gold-underline — not both).

Accent is NOT used for: links, all buttons, hover states, chart series fills generally, headings, borders, or "anything interactive". Gold is a factual marker only (matches vsizer `src/index.css` comment "factual marker, never an editorial flourish").

Chart series → token mapping (registered in `src/theme/echartsTheme.ts`, VIZ-03), light values; the theme module MUST expose a dark variant resolved from the same tokens:

| Chart element | Light token | Dark token | Notes |
|---------------|-------------|------------|-------|
| OS donut — Windows | `--color-primary-500` | `--color-primary-300` | Largest family anchors to brand navy |
| OS donut — Linux | `--color-primary-300` | `--color-primary-200` | |
| OS donut — Other | `--color-surface-200` | `--color-surface-700` | Neutral grey — "Other/unknown is a real, visible bucket" (never hidden) |
| Per-cluster stacked bar — Windows/Linux/Other | same 3 tokens as donut | same | One column per cluster; consistent OS encoding across donut and bar |
| Gauge CPU% / RAM% fill | `util-low` → `util-mid` → `util-high` banded by value | same | Bands at 0–60 / 60–80 / 80–100 (status only, no "good/bad" text) |
| Gauge — vCPU allocation ratio | `--color-primary-500` solid | `--color-primary-300` | A ratio, not a utilization — single brand color, NO red/green banding (avoids implying a verdict; Moderate-4) |
| Chart axis / grid / label text | `slate-500` / `slate-200` | `slate-400` / `surface-700` | Inherit theme; never pure black |
| Chart tooltip surface | `.panel` equivalent | `.panel` dark equivalent | Consistent with card surfaces |

Contrast: all text on `.panel` surfaces meets ≥4.5:1 in both themes (slate-900-on-white / slate-100-on-surface-800 are inherited Phase-1-verified pairings). Status colors are used as fills behind no text; the numeric value sits adjacent in body text color, not on the colored fill.

Layout rules:
- Region vertical rhythm: `gap-6` (24px) between toolbar → summary → donut → cluster grid → CPU Ready.
- `PerClusterColumns` is a single horizontal-scroll flex/grid row; each cluster is one fixed-min-width `.panel` column (`min-w-[200px]`, `gap-4` between columns). This is the RVTools-admin headline reading pattern (DSH-01) — do NOT reflow clusters into a vertical list.
- Per-cluster figure block: labels left (14px `.label`), values right-aligned in `--font-mono` `tabular-nums` so the same metric lines up vertically across all cluster columns when the user scans horizontally.
- Gauges inside a cluster column are small (≤96px) and captioned (12px). The estate-level OS donut is medium (≤320px height — matches `<Chart>` default `style={{height:320}}`).
- Single snapshot only this phase. The layout MUST NOT bake in single-snapshot assumptions that block Phase 6 sparklines-per-cluster (leave vertical room in the cluster column footer; `EstateView.trends` is `null` Phase 2).
- Empty/loading/error states replace the entire dashboard region (not the sidebar) — see Copywriting Contract.

## Number & Locale Formatting (surfaced to UI — required by `<must_specify>`)

All numbers rendered through the ported `utils/format.ts` (vsizer, "port unchanged"). NO pre-formatted numbers in i18n strings — translation strings carry `{{placeholders}}` only; the formatter is locale-aware off the active i18next locale (FND-03).

| Quantity | Format rule | EN example | FR example |
|----------|-------------|------------|------------|
| Counts (VMs, ESX, clusters, datastores) | Integer, locale thousands separator | `5,000` | `5 000` (U+202F/U+00A0 narrow no-break space) |
| CPU/RAM speed | GHz, 1 decimal, unit suffix shown | `62.4 GHz` | `62,4 GHz` |
| Memory / storage | GiB, 0–1 decimal, unit suffix shown (NEVER "GB" — RVTools MB is MiB, ADR-0010) | `1,024 GiB` | `1 024 GiO` |
| Utilization (CPU%/RAM%) | Percent, 1 decimal, `%` suffix | `73.2 %` | `73,2 %` |
| Allocation ratio | `n:1` form or `×`, 1–2 decimals; label reads "vCPU per physical core" (NOT "vCPU:pCPU" — ambiguous, Moderate-4) | `4.0 : 1` | `4,0 : 1` |
| CPU Ready | Percent, 2 decimals | `1.20 %` | `1,20 %` |
| Absent / not-applicable | Em-dash sentinel `—` (U+2014) via formatter's non-finite return — NEVER `0` or `N/A` text | `—` | `—` |

Unit suffixes (`GHz`, `GiB`, `%`, `:1`) are part of the value rendering, not the i18n label, so they are not re-translated and never drift. The em-dash sentinel is mandatory for `null` CPU Ready (DSH-05 / ADR-0012) and undeterminable per-cluster datastore counts (A1) — see CPU Ready semantics.

## CPU Ready & Status Color Semantics (DSH-05 — required by `<must_specify>`)

CPU Ready presentation per cluster and at estate roll-up:

| Field | Treatment |
|-------|-----------|
| Mean CPU Ready | Arithmetic mean (ADR-0012 §3 — not vCPU-weighted), 2-decimal `%`, body numeric |
| Max CPU Ready | Max via reduce (not spread), 2-decimal `%`, body numeric |
| Count > 5% | Integer count of VMs above `CONTENTION_THRESHOLDS.warning` (5%). This single number is the ONE accent-gold highlight allowed (see Color). Label: 14px; value: gold, 16px semibold mono. |
| Threshold color semantics | The 5% line is a FACTUAL VMware-documented warning threshold, surfaced as a neutral reference, NOT a verdict. Allowed: a thin neutral marker/tick at 5% on any CPU-Ready visual; the count colored gold (attention, not alarm). FORBIDDEN: words "good/bad/poor/healthy/warning/critical", red/green pass-fail framing on CPU Ready, or any "you should" copy (no editorial recommendations — PROJECT.md cultural rule, ADR-0003). |
| Not reported | When `readinessAvailable === false` (RVTools-only inputs frequently omit CPU Ready), render the literal localized "not reported" (12px micro, slate-500) — NEVER `0 %`, NEVER green, NEVER hide the row. This is the ADR-0012 §2 contract: absence ≠ zero ≠ healthy. |

CPU%/RAM% utilization gauges (DSH-03) use `util-low/mid/high` banding (0–60 / 60–80 / 80–100) as status-only fills with no judgment text — the band is a factual position indicator, identical posture to the shipped `.util-bar-fill` class. The vCPU-allocation gauge is single-color brand navy (a ratio is not a utilization; banding it would imply a verdict).

## Accounting-Mode Toggle (DSH-06 — required by `<must_specify>`)

| Property | Decision |
|----------|----------|
| Control type | Segmented control — 3 mutually-exclusive options in a single horizontal group. NOT a dropdown (all three must be visible — they are a core analytical lens, not a buried setting). Reuses the `aria-pressed` segmented pattern already shipped in Phase-1 `ThemeToggle.tsx` (DRY — same interaction language). |
| Options (labels) | "Configured" · "Active" · "Storage-realistic" — all three i18n-keyed (new `dashboard` namespace), localized FR/EN. |
| Placement | Dashboard toolbar, top-right, aligned with the "Estate summary" section title. Single instance — it governs the whole dashboard, not per-card. |
| Default state | `Active` (DSH-06 / Critical-6: Active is the sensible default for the CPU/RAM-led dashboard). `EstateView.accountingMode` is echoed back and shown as a 12px micro caption next to affected figures (e.g. summary card subtitle "(Active)") so the user always knows which lens is applied. |
| Active-mode visual signal | The selected segment uses `bg-primary-600 text-white` (filled) with `aria-pressed="true"`; unselected segments are `.panel`-neutral with `aria-pressed="false"`. Exactly one active-state treatment (filled navy) — do not also gold-underline (see Color accent list). Focus-visible ring `ring-2 ring-primary-500`. |
| Keyboard | Fully keyboard-operable (constraint: "the accounting-mode toggle … must be keyboard-operable"): Tab to the group, Arrow Left/Right (or Up/Down) move selection, Space/Enter activate; `role="group"` with an accessible group label "Accounting mode". Mirrors `ThemeToggle` keyboard behavior. |
| State location | UI state only — lifted `useState` at the dashboard root OR a `ui` store slice field (A3: a UI selection, not a dataset row — not a privacy concern, PAR-05 unaffected). `useEstateView(mode)` re-derives on change (one cheap memo miss; recompute, NOT precompute all three — KISS, Pitfall 2). |
| Mode meanings (surfaced as a tooltip/`aria-describedby`, factual) | Configured = all VMs incl. powered-off (vCPU/vRAM). Active = powered-on only. Storage-realistic = powered-on for CPU/RAM, all VMs for provisioned storage. Tooltip copy is descriptive only — no recommendation about which to pick. |
